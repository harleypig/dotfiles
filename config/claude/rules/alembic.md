# Alembic Rules

**Version:** v1.0.0

Conventions for Alembic database migrations. Pairs with `sqlalchemy.md`.

For the migration recipes autogenerate gets wrong (enum create/drop lifecycle,
data-backfill before NOT NULL, SQLite batch mode, downgrade FK order), invoke
the **sqlalchemy-patterns** skill.

## Detection

Active when an `alembic.ini` or an `alembic/` directory exists.

## Workflow

```bash
# After changing models:
alembic revision --autogenerate -m "describe the change"
# Review the generated script, then:
alembic upgrade head
```

- **Always review the autogenerated script before committing.**
  Autogenerate is a draft: it can miss server defaults, type changes,
  renames (it sees a drop + add), and custom types. Adjust by hand.
- Use a clear, imperative message; one logical change per revision.

## Writing migrations

- **Migrations are historical snapshots — do not import application
  models or app-specific types into them.** Use generic `sqlalchemy`
  types (e.g. `sa.String(36)` rather than a custom `GUID`). A migration
  must keep working even after the model code changes or moves.
- Enable batch mode for SQLite (`render_as_batch=True` in `env.py`) so
  `ALTER`-style changes work despite SQLite's limited `ALTER` support.
- Provide a real `downgrade` where practical.

## Scope

- Target **one database** (one declarative base's metadata) per Alembic
  environment. Regenerable databases (reseedable content, caches) do not
  need Alembic — `create_all` + reseed is fine (`sqlalchemy.md`).
- Resolve the database URL in `env.py` from the app settings, not a
  hard-coded `sqlalchemy.url`.

## Onboarding an existing database

- A database created before Alembic (tables via `create_all`, no
  `alembic_version`) must be **stamped** at the baseline before upgrading,
  so existing tables are adopted rather than recreated
  (`alembic stamp head` / `command.stamp`). Startup that auto-upgrades
  should auto-stamp this case.

## Agent Behavior

- Generate migrations with `--autogenerate`, then read and correct the
  script before committing — never commit an unreviewed autogenerated
  migration.
- Keep app imports out of migration scripts; use generic SQLAlchemy types.
- Ship the `alembic/` directory with the deployed app (e.g. copy it into
  the image) so `upgrade head` can run at startup.
