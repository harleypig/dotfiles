# SQLAlchemy Patterns

**Version:** v1.0.0

Concrete recipes for SQLAlchemy 2.0 (typed declarative) and Alembic. **The
conventions live in `rules/sqlalchemy.md` and `rules/alembic.md`** (typed
`Mapped`/`mapped_column`, one engine per logical DB, review autogenerated
scripts, no app imports in migrations) — read them first; this skill adds the
deeper *how*. The repository/service patterns are in the **fastapi-patterns**
skill; this skill owns models, sessions, queries, and migrations.

## Models — the depth beyond "use typed 2.0 style"

**Compose mixins** for the columns every table repeats — a UUID primary key
and created/updated timestamps — so models stay about their own fields:

```python
import uuid
from datetime import datetime
from sqlalchemy import func
from sqlalchemy.orm import Mapped, mapped_column

class UUIDPrimaryKey:
    id: Mapped[uuid.UUID] = mapped_column(
        primary_key=True, default=uuid.uuid4
    )

class Timestamps:
    created_at: Mapped[datetime] = mapped_column(server_default=func.now())
    updated_at: Mapped[datetime] = mapped_column(
        server_default=func.now(), onupdate=func.now()
    )

class User(UUIDPrimaryKey, Timestamps, Base):
    __tablename__ = "users"
    email: Mapped[str] = mapped_column(unique=True, index=True)
```

**Encapsulate custom storage once** with a `TypeDecorator` — e.g. storing a
UUID as a 36-char string on SQLite — and reuse it, rather than scattering
conversion logic:

```python
from sqlalchemy import String, TypeDecorator

class GUID(TypeDecorator):
    impl = String(36)
    cache_ok = True

    def process_bind_param(self, value, dialect):
        return None if value is None else str(value)

    def process_result_value(self, value, dialect):
        return None if value is None else uuid.UUID(value)
```

**Relationships** declare both ends with `back_populates` and an explicit
`order_by` when order matters:

```python
from sqlalchemy.orm import relationship

class User(Base):
    orders: Mapped[list["Order"]] = relationship(
        back_populates="user", order_by="Order.created_at"
    )
```

## Sessions and the unit of work

The session dependency yields one session per request and closes it; **the
commit belongs to the unit of work, not the repository** (repos `flush`). A
service method that touches several repositories commits once at the end, so
the whole operation is atomic:

```python
from sqlalchemy.orm import Session

def get_session():
    session = SessionLocal()
    try:
        yield session
    finally:
        session.close()

# service — one transaction across two repositories
def place_order(self, data: OrderCreate) -> Order:
    order = self._orders.add(Order(user_id=data.user_id))   # flush, no commit
    for item in data.items:
        self._items.add(OrderItem(order_id=order.id, **item.model_dump()))
    self._session.commit()    # single boundary
    return order
```

For SQLite, install the FK pragma listener (`rules/sqlalchemy.md`) — foreign
keys are off per-connection otherwise.

## Killing N+1 — eager loading

Lazy relationships fire one query per parent on access. Load them up front:

- **`selectinload`** — a second `IN (...)` query; best for collections
  (one-to-many). Default choice.
- **`joinedload`** — a single `JOIN`; best for many-to-one / one-to-one.
  Avoid on collections (row fan-out).

```python
from sqlalchemy.orm import selectinload

stmt = select(User).options(selectinload(User.orders)).where(User.id == id_)
user = session.execute(stmt).scalar_one_or_none()
```

If a list endpoint feels slow, the cause is almost always a missing
`selectinload` — check there before optimizing anything else.

## Alembic migration recipes (the value-add)

`rules/alembic.md` already says *review the autogenerated script*. These are
the specific things autogenerate gets wrong that you have to fix by hand.

**Enum lifecycle — the #1 autogenerate gap.** Autogenerate creates a named
enum type in `upgrade` but does **not** drop it in `downgrade`, orphaning the
type. Manage it explicitly, with `checkfirst` to stay idempotent:

```python
import sqlalchemy as sa

status = sa.Enum("pending", "shipped", name="order_status")

def upgrade() -> None:
    status.create(op.get_bind(), checkfirst=True)        # before the column
    op.add_column("orders", sa.Column("status", status, nullable=False))

def downgrade() -> None:
    op.drop_column("orders", "status")
    status.drop(op.get_bind(), checkfirst=True)          # after the column
```

When an enum is shared by several tables, check it is unused before dropping
it in any one table's downgrade. (`DuplicateObject` on upgrade → you skipped
`checkfirst`; `DependentObjectsStillExist` on downgrade → you dropped the type
before the column.)

**Adding a NOT NULL column to existing rows** needs a data backfill *between*
add-nullable and set-not-null, or the migration fails on existing data:

```python
def upgrade() -> None:
    op.add_column("users", sa.Column("role", sa.String(), nullable=True))
    op.execute("UPDATE users SET role = 'member' WHERE role IS NULL")
    op.alter_column("users", "role", nullable=False)
```

**SQLite needs batch mode** — it has almost no `ALTER` support, so wrap
alterations; set `render_as_batch=True` in `env.py` and use
`batch_alter_table` for column changes:

```python
with op.batch_alter_table("users") as batch:
    batch.alter_column("email", existing_type=sa.String(), nullable=False)
```

**Downgrade dependency order** — drop FK constraints and child tables before
the parent they reference, the mirror of the upgrade order.

## Troubleshooting

When a model or migration misbehaves in a non-obvious way — a `Mapped`
annotation that won't map, a relationship that won't load, a migration that
runs but leaves the schema wrong — **run the `qa-check` skill** before deep
debugging. The format/lint/type-check pass catches the common causes (a
mistyped `Mapped[...]`, an import that resolves to the wrong base, a column
type mismatch) quickly. Fix what it surfaces, then resume.
